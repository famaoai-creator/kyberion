# SU-04: ミッション履歴検索・コスト可視化・承認キュー

> 優先度: P2 / 規模: M / 依存: OP-01(コスト)、IL-02(相関)、SU-01(ホーム) / 関連: POST_ONBOARDING_UX_ROADMAP #2/#3
>
> **なぜ重要か**: 「先週のあのレポートを探す」「これはいくらかかった」「保留中の承認をまとめて捌く」— 日常運用の基本操作が UI に無い、または CLI 限定。

## 背景と課題

- **履歴検索が無い**: chronos は**アクティブなミッションのみ**表示(`api/intelligence/route.ts:615` が `status==='active'` でフィルタ)。完了/履歴の閲覧も検索も無い。operator-surface は全 status を出すが read-only かつ検索不可。
- **コスト可視化がゼロ**: token/コスト/spend のフィールドがどちらの surface のペイロード・コンポーネントにも存在しない(OP-01 のバックエンドが計測しても UI が無い)。
- **承認キューが埋没**: `approval_decision`(approve/reject)は MissionIntelligence のサブパネルに埋め込まれ(`:1598-1634`)、独立した承認トリアージキューが無い。リクエストの**修正**もできない(approve/reject のみ)。
- **接続レビュー/テナントバナーが無い**(POST_ONBOARDING #2/#3): 検出された各接続を理解し承認/保留する画面、操作前に「今どのテナント文脈か」を示す常時バナーが、どちらの surface にも無い。

## ゴール(受入条件)

1. **ミッション履歴の検索・閲覧**: 完了含む全ミッションを検索(キーワード・tenant・日付・status・成果物種別)でき、相関 ID(IL-02)で intent→result を辿れる。
2. **コスト可視化**: ミッション別・当日/期間のコスト(OP-01 の集計)がホーム(SU-01)とミッション詳細に表示される。
3. **承認キュー**: 全保留承認を 1 画面でトリアージでき、各リクエストの帰結(SA-05/UX-04 の「承認/待つ/拒否の結果」)を見て判断・(可能なら)修正できる。
4. **テナントコンテキストバナー**: 操作前に現在のテナント文脈が常時見える(POST_ONBOARDING #3)。**接続レビュー画面**(#2)で検出接続を承認/保留。

## 実装タスク

### Task 1: ミッション履歴・検索 — `claude-sonnet-4`

1. `api/intelligence` のミッション一覧を active 限定(`:615`)から「全 status + フィルタ/検索」に拡張(または別 API `app/api/missions/search`)。検索インデックスは mission state + 成果物メタ + 相関(IL-02)。
2. 履歴一覧 UI(SU-01 ホームからリンク): 検索ボックス、フィルタ、ミッション → intent(元発話)→ 成果物への導線。
3. テスト: 完了ミッションの検索・フィルタ。

### Task 2: コスト可視化 — `claude-sonnet-4`

1. OP-01 の `cost report` 集計を API 化(`app/api/cost`)し、ホーム(当日累計)・ミッション詳細(当該ミッションの token/USD・内訳)に表示。
2. 予算上限(OP-01 の spend-guard)の状態(残り/超過)も表示。
3. テスト: fixture メトリクスからの UI 表示。

### Task 3: 承認キュー — `claude-sonnet-4`

1. 独立した承認トリアージ画面: 全保留承認(approval-store)を一覧、各々に SA-05/UX-04 の帰結情報(承認/待つ/拒否の結果、リスク)を表示、approve/reject/(可能なら)修正。
2. MissionIntelligence 埋め込みの `approval_decision`(`:1598-1634`)はこのキューへ集約(重複 UI を解消)。
3. テスト: キューでの一括トリアージ。

### Task 4: テナントバナー・接続レビュー — `claude-sonnet-4`

1. 常時テナントコンテキストバナー(POST_ONBOARDING #3): 現在の tenant/customer 文脈を全画面ヘッダに表示(DS-02 の tenant 解決と整合、切替時に明示)。
2. 接続レビュー画面(#2): 検出された service 接続を一覧し、approve/modify/hold/delete。onboarding(ONB)の service フェーズと接続。
3. POST_ONBOARDING_UX_ROADMAP #2/#3 に「SU-04 として実装」とステータス追記。
4. テスト: バナーがテナント文脈を反映、接続レビューの各操作。

## リスクと注意

- 履歴検索は完了ミッションが増えると重くなる。ページング + インデックス(相関 ID・日付)を最初から入れる。KM-01 の janitor でアーカイブ済みミッションの扱いを定義。
- コスト表示は confidential ミッションの内訳を tenant 文脈外に出さない(tier 隔離)。
- 承認キューの「修正」機能は SA-05 の承認ポリシーと整合させる(修正が新たな承認を要する場合の扱い)。
