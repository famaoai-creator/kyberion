# 📖 Kyberion HOWTO: Operational Guide

This document defines the standard procedures for interacting with the Kyberion Sovereign Ecosystem.

---

## 🎯 1. Running Commands (The Intent Gateway)

In Kyberion v2.1+, you do not run individual scripts. Instead, you issue **Intents**. The Intent Resolver will automatically map your request to the correct cross-actuator pipeline.

### Standard Intent Execution
```bash
# Recommended way to run any high-level task
node dist/scripts/run_intent.js <intent_id>
```

### Manual Pipeline Execution
If you need to run a specific ADF pipeline file:
```bash
node dist/scripts/run_super_pipeline.js --input path/to/pipeline.json
```

## 🧠 2. Mission Workflow (KSMC)

All physical changes to the codebase MUST be performed within a mission context.

### Full Lifecycle

```
start → checkpoint (repeat) → verify → distill → finish
```

1. **Start**: `node dist/scripts/mission_controller.js start <ID> --tier <tier> [options...]`
   Creates and activates a mission. Status: `planned → active`.
   Only `<ID>` is positional. Everything else is a named option.
   Example:
   `node dist/scripts/mission_controller.js start TASK-01 --tier confidential --persona ecosystem_architect --project-id PRJ-1 --project-path active/projects/sample --track-id TRK-REL1`

2. **Work & Checkpoint**: Perform tasks, then record progress:
   `node dist/scripts/mission_controller.js checkpoint <ID> <TASK_ID> "description"`

3. **Verify**: Mark the mission as verified (or reject for rework):
   `node dist/scripts/mission_controller.js verify <ID> verified "Verification note"`
   Status: `active → distilling` (or back to `active` if rejected).

4. **Distill**: Extract knowledge from the mission via LLM:
   `node dist/scripts/mission_controller.js distill <ID>`
   Status: `distilling → completed`. Produces a wisdom file in `knowledge/`.

5. **Finish**: Archive the completed mission:
   `node dist/scripts/mission_controller.js finish <ID> [--seal]`
   Optionally encrypts with `--seal` (AES+RSA).

### Project Operational State

When a mission is linked to a project, Kyberion keeps the live operational truth under:

`active/projects/<tier>/<tenant_or_shared>/<project_id>/state/`

- Mission, track, and task-session links are recorded there as raw operational state.
- `project-os/` remains the scaffolded project document set.
- Distillation converts that raw state into reusable knowledge under `knowledge/product/evolution/` or `knowledge/product/incidents/`.

### Visibility

```bash
# List all missions (optionally filter by status)
node dist/scripts/mission_controller.js list [active|completed|...]

# Show detailed status of a specific mission
node dist/scripts/mission_controller.js status <ID>

# Show all commands
node dist/scripts/mission_controller.js help
```

## 📬 3. A2A Communication

To communicate with Kyberion via the **Agent-to-Agent (A2A)** protocol:

```bash
node dist/scripts/run_a2a.js --input my-a2a-message.json
```

### Mission Collaboration
Kyberion uses a `single-owner, multi-worker` mission model.

- The owner agent controls mission state and lifecycle.
- Worker agents operate through task contracts and leases.
- Shared execution artifacts should go to mission-local `coordination/` first, and only use `active/shared/` for global discovery or observability.

The response will be a standardized A2A `result` message containing the pipeline context.

## 🧭 3.5 Control Plane CLI

Use the control-plane CLI when you want the same project/track/gate flow that Presence Studio and Chronos expose:

```bash
pnpm control presence tracks [projectId]
pnpm control chronos tracks [projectId]
pnpm control chronos mission-seeds
pnpm control chronos ref <knowledge/...|active/projects/...>
pnpm control chronos seed-track <trackId> [artifactId]
```

Mental model:

```text
Project -> Track -> Gate Readiness -> Next Required Artifact -> Template/Skeleton -> Mission Seed -> Mission
```

## 🛠 4. Actuator Reference

Kyberion's "body" consists of several universal actuators, each specialized in a domain but controlled via the same pipeline engine.

| Domain | Actuator Path | Responsibility |
| :--- | :--- | :--- |
| **System** | `libs/actuators/system-actuator` | OS, QA, Governance, Pulse |
| **Wisdom** | `libs/actuators/wisdom-actuator` | Knowledge Sync, Distillation, Metadata |
| **File** | `libs/actuators/file-actuator` | Secure I/O, Search, Tail, FS Ops |
| **Network** | `libs/actuators/network-actuator` | External API (A2A), Ingestion |
| **Browser** | `libs/actuators/browser-actuator` | Playwright-driven Automation |
| **Orchestrator** | `libs/actuators/orchestrator-actuator` | Mission Mgmt, Deploy, Setup |

## 🛡 5. Safety & Security

- **Safe-IO**: Always use `@agent/core` wrappers. Direct `node:fs` is prohibited.
- **SecretGuard**: Never log raw API keys. The system masks them automatically.
- **Tier Isolation**: Respect the 3-Tier model (Personal, Confidential, Public).
