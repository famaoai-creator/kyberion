# 自律運用・保守の判断基準 — Fable の考え方を opus/sonnet/haiku で再現する

> **作成日**: 2026-07-03
> **目的**: Kyberion に長時間の運用・保守を任せる際、「何を自動で行い・何で人間を呼び・脆弱性パッチをいつ当てるか」を、**モデルに依らず再現できる決定論的な判断基準(ルーブリック)**として文書化する。Fable が個体として持つ判断を、opus/sonnet/haiku、さらに軽量な backend でも同じ結論に至れるよう、点数化・閾値化する。
> **関係文書**: 個体の振る舞い原則は [FABLE5_AGENT_MODEL](./FABLE5_AGENT_MODEL.ja.md)、指揮法は [ORCHESTRATION_HARNESS_MODEL](./ORCHESTRATION_HARNESS_MODEL.ja.md)。本文書はそのうち**運用・保守の判断**を実務ルーブリック化したもの。実装は [improvement-plans-2026-07](./improvement-plans-2026-07/README.ja.md) の OP/AO/SA 系。

---

## 0. なぜルーブリックにするか

Fable は「可逆なら進め、破壊的・不可逆・外向きは確認する。完了を自分でも疑う。不確実は不確実として運ぶ」という判断を内面化している。だが**軽量モデルはこの判断を安定して再現しない**(§FABLE5_AGENT_MODEL の通り、強いモデルは雑な判断を補完するが軽量は補完しない)。

したがって運用・保守のように「毎日・無人で・繰り返す」判断は、モデルの内面化に頼らず**明示的なルーブリック(入力 → 点数 → 閾値 → アクション)**にする。ルーブリックが決定論なら、どのモデルが実行しても同じ結論になり、監査でき、閾値を運用で調整できる。**これが「Fable の考え方を opus/sonnet/haiku で運用する」ことの実体**である。

---

## 1. 全アクションを貫く 4 軸(自動 or 承認の基準)

保守アクションを実行してよいかは、次の 4 軸で決める。各軸を 0–3 で採点する。

| 軸                               | 0(低)                       | 3(高)                        |
| -------------------------------- | --------------------------- | ---------------------------- |
| **可逆性 (Reversibility)**       | 1 コマンドで元に戻せる      | 元に戻せない/データ喪失      |
| **影響範囲 (Blast radius)**      | 単一ファイル/単一ミッション | システム全体/全テナント/外部 |
| **判定信頼度 (Confidence)** の逆 | 決定論的に確実              | LLM 判断で不確実             |
| **外部性 (Externality)**         | ローカルのみ                | 外部送信/公開/課金発生       |

**判定ルール**(合計点でなく最大軸で決める — 安全側):

- **どの軸も ≤1** → **自動実行可**(記録は必須)。
- **いずれかの軸が 2** → **自動実行 + 事後通知**(実行するがオペレータに要約を送る)。
- **いずれかの軸が 3** → **事前承認必須**(実行前に人間へ。SA-05 の承認ゲート)。
- **可逆性=3 かつ 影響範囲≥2** → **常に承認必須**(不可逆 × 広域は例外なく人間)。

この 4 軸は Fable の「可逆は進め、破壊的・外向きは確認」を数値化したもの。Kyberion では SA-05 の承認ポリシー・OP の spend-guard・AO の自律保守ループの判定源として共有する。

## 2. 定常保守アクションの分類(既定の自律度)

日常保守の各アクションに、§1 で導かれる既定の自律度を割り当てる。これを `knowledge/product/governance/autonomous-ops-policy.json` 等に外出しし、テナント override 可能にする。

| アクション                              | 可逆/範囲/信頼/外部   | 既定                                                     |
| --------------------------------------- | --------------------- | -------------------------------------------------------- |
| tmp/ログの TTL GC(KM-01)                | 0/0/0/0               | **自動**                                                 |
| 揮発メモリの GC/rollover(KM-01)         | 1/0/0/0               | **自動**                                                 |
| 健全性スキャン・doctor(OP-04)           | 0/0/0/0               | **自動**                                                 |
| 依存の**patch**更新(x.y.Z)+ テスト緑    | 1/1/0/0               | **自動 + 事後通知**                                      |
| 依存の**minor**更新(x.Y.z)              | 2/1/1/0               | **自動 + 事後通知**(テスト緑が条件)                      |
| 依存の**major**更新(X.y.z)              | 2/2/1/0               | **承認必須**                                             |
| クラッシュしたランタイムの再起動(AA-01) | 1/1/0/0               | **自動 + 事後通知**(再起動ループ上限内)                  |
| stale リース/セッションの cleanup       | 1/1/0/0               | **自動 + 事後通知**                                      |
| ミッションデータのアーカイブ            | 1/1/0/0               | **自動 + 事後通知**                                      |
| バックアップ作成(OP-02)                 | 0/1/0/1(外部先)       | **自動 + 事後通知**                                      |
| **脆弱性パッチ適用**                    | §3 の専用ルーブリック | §3                                                       |
| 設定/ポリシーファイルの変更             | 2/3/1/0               | **承認必須**                                             |
| シークレット/鍵のローテーション         | 3/2/1/1               | **承認必須**                                             |
| kill-switch の kill 発火(SA-05)         | 3/3/2/0               | **承認必須**(warn/isolate は自動)                        |
| 外部への送信・公開                      | —/2+/—/3              | **承認必須**(egress allowlist 内の定常送信を除く、SA-04) |

原則: **低リスクの衛生作業は無人で回し、不可逆・広域・設定変更・鍵・kill は必ず人間を通す。** 迷ったら承認側に倒す(fail-closed、SA-05 と同じ姿勢)。

## 3. 脆弱性パッチ適用の判断ルーブリック(専用)

脆弱性パッチは「当てないリスク」と「当てて壊すリスク」の両方があるため、専用の点数化で判断する。

### 3.1 緊急度スコア(当てるべき強さ)

```
urgency = severity_weight × reachability × exposure
```

- **severity_weight**(CVSS ベース): Critical=4 / High=3 / Medium=2 / Low=1。
- **reachability**(その脆弱コードが実際に到達可能か): 実行経路に含まれる=2 / 依存の依存で間接=1 / 使っていない機能=0。**「schema 検証を通っても入力は信頼できない」の姿勢で、疑わしきは到達可能側に寄せる。**
- **exposure**(攻撃面): 外部入力/ネットワーク面=2(ブラウザ・サービス・ブリッジ経由)/ ローカルのみ=1 / オフライン=0。

### 3.2 適用リスクスコア(当てて壊す危険)

```
apply_risk = semver_jump × test_gap × rollback_difficulty
```

- **semver_jump**: patch=1 / minor=2 / major=3。
- **test_gap**(その依存を使う経路のテスト網羅): 十分=1 / 部分=2 / ほぼ無し=3(IP-03 の CI 網羅と連動)。
- **rollback_difficulty**: lockfile revert で戻せる=1 / 移行を伴う=2 / データ/スキーマ変更を伴う=3。

### 3.3 判定マトリクス

| urgency | apply_risk | アクション                                                                                             |
| ------- | ---------- | ------------------------------------------------------------------------------------------------------ |
| 高(≥12) | 低(≤4)     | **自動適用 + テスト + 事後通知**(すぐ当てて安全)                                                       |
| 高(≥12) | 高(≥12)    | **緊急承認要求**(当てるべきだが壊す危険 — 人間が判断、緩和策を併走: 該当機能の一時無効化・egress 制限) |
| 低(≤6)  | 低(≤4)     | **次の定期メンテでまとめて自動適用**                                                                   |
| 低(≤6)  | 高(≥12)    | **見送り + 監視**(当てる利益 < 壊す危険。CVE を追跡し状況変化で再評価)                                 |
| 中間    | 中間       | **承認要求**(既定は人間に判断材料を添えて委ねる)                                                       |

### 3.4 適用フロー(自動適用が選ばれた場合)

必ずこの順で、各段で失敗したら停止・ロールバック・エスカレート:

1. **バックアップ**(OP-02、lockfile + 影響範囲の状態)
2. **隔離ブランチ/作業ツリーで適用**(本番作業ツリーを直接触らない)
3. **ビルド + 該当経路のテスト**(IP-03、無ければ最低限のスモーク)
4. **脆弱性再スキャン**(当該 CVE が解消したか、AO-02)
5. **緑なら適用確定 + 監査記録(SA-01)+ 事後通知**、赤なら**ロールバック + 承認要求**(「自動適用に失敗、手動判断が必要」)
6. 適用後の**カナリア監視**(OP-04 の劣化検知で回帰がないか一定時間観測)

**当てない判断も記録する**(§3.3 の見送り)。無音で放置しない — CVE・理由・再評価条件を追跡台帳に残す(Fable の「落としたものを log する」原則)。

## 4. 運用の判断ループ(観測 → 診断 → 判断 → 実行 → 検証 → 記録)

無人運用は次の決定論ループを回す。各段は前段の構造化出力を受ける(HN-02 の schema-forced)。

1. **観測**: 定期(chronos cron)で health/コスト/劣化/脆弱性/バックログを収集(OP-04/OP-01/AO-02)。
2. **診断**: 収集値を閾値・ルーブリックに照合し、問題を分類(「劣化」「脆弱性」「リソース逼迫」「クラッシュ」等)。**LLM でなく決定論の閾値**を第一段にし、曖昧な分類だけモデルに委ねる。
3. **判断**: §1–§3 のルーブリックで各問題に自動/通知/承認を割り当てる。
4. **実行**: 自動分は実行(可逆・記録付き)、承認分は人間へエスカレート(§5)。
5. **検証**: 実行結果を動かして確認(Fable の「動くのを見た」)。回帰があればロールバック。
6. **記録**: 監査(SA-01)+ 学習(KM-03、「よく起きる問題と有効だった対処」を昇格)。

このループ自体が「Fable の考え方」の運用版: 決定論の骨格に、判断が要る所だけモデルを挿す(HN-03)。

## 5. エスカレーションの判断 — いつ・どう人間を呼ぶか

- **呼ぶべき時**: §1 で承認必須と出た時 / 自動実行が失敗しロールバックした時 / 同種問題が閾値回数を超えて再発する時(自己修復が効いていない兆候)/ 予算(OP-01)や劣化(OP-04)が危険域 / ルーブリックで分類できない未知の問題。
- **呼び方(Fable の「読み手のための報告」)**: bare な通知でなく、**判断材料を添える** — 何が起きたか / 影響 / 自動で試したこと / 選択肢と推奨 / 待つ/承認/拒否した場合の帰結(HO-01 のリッチ承認 framing、UX-04 の帰結表示)。人間が数十秒で判断できる形。
- **呼びすぎない**: 低リスクを逐一承認要求するとアラート疲れで承認が形骸化する。§2 の分類で自動範囲を十分に広げ、人間は本当に判断が要る所だけに集中させる。
- **多重通知の抑制**: 同一問題の連続エスカレーションはレート制限(UX-01 の連投抑制と同じ)。

## 6. 長時間運用のための不変条件(無人で守るべき線)

自動化がどれだけ進んでも、無人運用中に**破ってはならない線**:

1. **不可逆 × 広域は必ず人間**(§1)。自動化が「効率のため」にこの線を越えない。
2. **fail-closed を維持**(SA-05)。判断できない・ポリシーが読めない時は、実行でなく承認要求に倒す。
3. **すべての自律アクションを監査に残す**(SA-01)。無人だからこそ後で追える必要がある。
4. **予算のハードシーリング**(OP-01)。無人で暴走コストを出さない。
5. **kill-switch は生きている**(SA-05)。異常検知でエスカレート/隔離できる状態を常に保つ。
6. **落としたもの・見送ったものを記録**(§3.4)。無音の放置をしない。

## 7. モデル別の運用(opus/sonnet/haiku で同じ判断に至る)

ルーブリックが決定論なら、判断の**大部分はモデルに依らない**(点数化された閾値判定)。モデルが関与するのは「曖昧な分類」「未知問題の診断」「エスカレーション文面の生成」など少数のノード。そこでのモデル選択(HN-01 の tier):

- **haiku**: 定型の観測・分類・ルーブリック採点(決定論部分の実行)・定型通知。**判断ノードでは使わない**。
- **sonnet**: 標準的な診断・自動修復の実行・パッチ適用フローの遂行。
- **opus**: 未知問題の診断、脆弱性の到達可能性評価(誤ると危険)、緊急 × 高リスクパッチの判断材料生成、エスカレーション時の推奨。

重要なのは、**判断の正しさをモデルの賢さに賭けないこと**。ルーブリックを第一段に置き、モデルは「材料の整理と曖昧部分の補助」に留める。これにより haiku 中心の安価な無人運用でも、Fable と同じ結論に安定して至る。

---

## 8. 一文で

**長時間の自律保守とは「Fable の判断(可逆は進め・破壊は確認・完了を疑い・落としたものを記録)を決定論ルーブリックに落とし、観測→診断→判断→実行→検証→記録のループで無人で回すこと」。** ルーブリックが決定論だから opus/sonnet/haiku のどれでも同じ判断になり、監査でき、線を守れる。

→ これを回すために必要な機構の実装は AO 系(自律保守ループ・依存/脆弱性管理・無人耐久性)と、既存の OP/SA/AA/KM 系。
