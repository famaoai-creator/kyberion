## 0. 中心的な考え方(一段落で)

**最高の成果物は、単発の賢いモデルからでなく、"やり方(ハーネス)" から生まれる。** モデルは半年で置き換わる可換な部品であり、価値の源泉は「タスクをどう分解し・誰に・どんな依頼で・どの順で渡し・成果をどう評価し・どう改善ループを回し・どう引き継ぐか」という**決定論的な進め方**にある。これは Kyberion 自身のテーゼ(INTENT_LOOP_CONCEPT の「不可換の核 vs 可換の周辺」)と完全に一致する。したがって Kyberion が取り込むべきは特定モデルでなく、**軽量モデルでも高品質な成果を出せるオーケストレーションの型**である。

軽量モデル(haiku 級)をサブエージェントに使うほど、この型の質が成果を左右する。**強いモデルは雑な依頼でも補完するが、軽量モデルは補完しない。** ゆえに「ブリフの自己完結度・スコープの狭さ・出力の構造化・検証の厳格さ」を上げることが、軽量モデル活用の中心的技術になる。

## 1. タスク分解 — scout してから fan-out する

### 原則

- **計画とは「作業リストの発見」である。** 実行の前に、まず対象を軽く走査(scout)して作業単位を確定させ、それから並列展開する。作業の形が分かる前に fan-out しない。
- **型が実行の形を決める。** タスクの種類(理解・設計・実装・レビュー・機械的変換)ごとに、相・ゲート・既定編成・必要な検証が決まる。型が決まればプロセスは決定論的に導ける。
- **1件目でパターンを確立し、2件目以降で横展開する。** パターン確立は上位モデル(判断)、確立後の横展開は軽量モデル(機械的)に割り当てる。本改善計画群自体がこの型で書かれている(sonnet でパターン、haiku で横展開)。
- **分解の粒度**: 各タスクは「独立に検証可能な 1 つの成果」に対応させる。依存は明示し、同一ファイルを触るタスクには依存を張る(競合の直列化)。

### Kyberion の現状

- ミッションの型分類(9 クラス)は計算されるが実行は汎用テンプレに潰れる。タスク分解は planner LLM の 1 ターンに全依存し、`dependencies`/`acceptance_criteria` を持たない構造で永続化される(調査: MO-01/MO-03 参照)。
- scout→fanout の型は無く、planner 出力の `next_tasks` をそのまま逐次処理する。

### 取り込み(対応計画)

- **MO-01**(ミッションタイプの実効化・プロセステンプレート駆動)、**MO-03**(タスク契約に依存/受入条件、DAG 並列分配)、**HN-01**(型別のモデル階層割当)。

## 2. 依頼内容 — 自己完結したブリフを書く

### 原則

- **サブエージェントは会話履歴を知らない。** 依頼は 1 プロンプトで自己完結させる: 目的・背景・制約・成果物の定義(受入条件)・**返答形式**・エスカレーション条件・「分からなければ推測せず報告せよ」。
- **渡すのは要約された必要十分な事実であり、生ダンプではない。** コンテキストには予算(文字数)を切り、超過分はロールアップ要約に逃がす。redaction で見せてはいけない層を明示する。
- **返させるのは結論と根拠であり、ファイル全文や作業ログではない。** 構造化スキーマで返させ、検証してから受理する。
- **軽量モデルほどこの規律を強める**: スコープをさらに狭く、受入条件をより具体的に、出力形式をより厳格に、そして返ってきたものをより厳しく検証する。

### Kyberion の現状

- 優秀な `MissionContextPack`(予算 6000 字・redaction・「推測せずギャップ報告せよ」で終わる)が**存在するのに dispatch で使われず**、worker には手組みの薄い文字列が渡る(調査: MO-04)。
- 返答は自由テキストで、best-effort な JSON ブロック抽出 + 手組み構造アサートのみ。`a2a-task-contract.schema.json` はあるが return-path で使われない(調査: HN 確認 §5)。

### 取り込み(対応計画)

- **MO-04**(context pack の dispatch 配線・構造化結果契約)、**HO-01**(自己完結ハンドオフパケット・契約に rationale/受入条件)、**HN-02**(schema-forced 委譲と出力の契約検証)。

## 3. タスク順序 — 既定はパイプライン、バリアは横断参照の時だけ

### 原則

- **多段作業の既定はアイテム独立パイプライン**。アイテム A がステージ 3 にいる間、アイテム B はステージ 1 でよい。壁時計時間は「最も遅い 1 アイテムの連鎖」であって「ステージごとの最遅の総和」ではない。
- **バリア(全員待ち合わせ)は「全アイテムの結果を横断して使う」時だけ**正当化される(全件 dedup、ゼロ件での早期終了、"他の発見" との比較)。「flatten/filter したいから」「概念的に別ステージだから」「コードが綺麗だから」はバリアの理由にならない。
- **依存の無いタスクを直列に待つのは純粋な無駄。** 並列度キャップ(同時実行数の上限)を付けてキューイングする。
- **決定性のため順序を固定**(再実行の再現性): ready 集合内は id 昇順など。

### Kyberion の現状

- タスク分配は `for … await` の完全直列で、DAG も並列も無い。`max_parallel_members` は宣言されるが dispatcher が読まない。ADF パイプラインの `core:foreach` も**逐次実行**で、並列/map op が無い(調査: MO-03、HN 確認 §3)。

### 取り込み(対応計画)

- **MO-03**(DAG 並列分配・直列 for ループ脱却)、**HN-03**(パイプライン言語に並列 foreach/map、loop-until を追加)。

## 4. 成果物の評価 — 完了主張を信用せず、動かして・反証させる

### 原則

- **「完了した」を信用しない。成果物を実際に動かして観測する(verify)。** テストや型チェックが緑なだけでなく、影響する経路を駆動して挙動を見る。
- **敵対的検証**: 重要な発見・成果はデフォルトで疑い、独立した検証者に**反証を試みさせる**。多数決(N 人の懐疑者、過半が反証したら棄却)。
- **多角検証**: 成果が複数の様式で失敗し得るなら、検証者ごとに異なるレンズ(正しさ・セキュリティ・再現性)を与える。冗長でなく多様性で失敗様式を捕らえる。
- **judge panel / best-of-N**: 解空間が広い時は、異なる角度から N 個生成し、並列 judge で採点し、勝者から統合(次点の良い所も接ぎ木)。
- **スケールを要求に合わせる**: 「バグを探して」は少数の finder + 単票検証。「徹底監査して」は大規模 finder プール + 3〜5 票の敵対的検証 + 統合。

### Kyberion の現状

- 自動フローに独立レビュアーも敵対的検証も「成果物を動かす」検証も無い。完了は自己申告の `NEXT_TASKS.json` ステータスを信用する(調査: MO-02)。
- **decision 領域には本格的な品質機構(best-of-N アンサンブル・cross-critique・品質ルーブリック)が実装済みなのに、deliverable 生成には一切配線されていない。** しかも品質 severity(`poor`=「上位モデルで再実行せよ」)を計算するのに誰も consume しない dead-end(調査: MO-07)。

### 取り込み(対応計画)

- **MO-02**(計画ゲート・受入ゲート・敵対的レビュー・circuit breaker)、**MO-07**(休眠品質機構の起動・best-of-N・judge・敵対レビュー・draft-refine・品質ゲート付き redo)、**IL-04**(完了を元 intent に突き合わせる)。

## 5. 改善ループ — 枯れるまで回し、低品質なら上げる

### 原則

- **loop-until-dry**: 未知サイズの発見(バグ・エッジケース・不足)は、K 回連続で新発見ゼロになるまで finder を回す。単純な回数上限は末尾を取りこぼす。
- **completeness critic**: 最後に「何が漏れているか(未実行の様式・未検証の主張・未読の情報源)」を問うエージェントを置き、その発見を次ラウンドの作業にする。
- **retry-until-quality / escalation**: 成果物が品質基準を下回ったら、より強いモデル/大きい予算で再試行する。固定回数の transient リトライとは別物。
- **draft → critique → revise**: 長文成果物は一発生成でなく、下書き→ルーブリック自己批評→改稿(上限付き)。
- **accumulate-to-target**: 「バグ 10 件見つかるまで」のような目標到達までの累積ループ。
- **収束の判定は「見た集合(seen)」に対して行う**。承認済みでなく seen に対して dedup しないと、棄却された発見が毎ラウンド蘇って収束しない。

### Kyberion の現状

- loop-until-quality / accumulate-to-target / escalation-on-failure は**存在しない**。リトライは固定 2 回の transient エラーのみ、品質ベースでない(調査: HN 確認 §6)。review フェーズの学習(distill)は手動昇格で閉じていない(調査: KM-03)。

### 取り込み(対応計画)

- **MO-07**(品質ゲート付き redo・escalation・draft-refine)、**MO-02**(circuit breaker による再突入)、**KM-03**(学習ループの閉包)、**HN-03**(loop-until / retry-until-quality をパイプライン言語に)。

## 6. エージェント間コミュニケーション — 構造化・自己完結・追跡可能

### 原則

- **メッセージは構造化された契約**で運ぶ(自由テキストでなく)。送信も返答もスキーマで、検証してから受理。
- **ハンドオフは自己完結**: 次の holder が聞き返さず継続できるだけの情報(何をしたか・未決事項・部分成果物・残受入条件・**なぜ**)を渡す。
- **相関 ID で貫通**: 発話 → ゴール → タスク → 成果 → 検証を 1 本の ID で辿れる。
- **配送保証と耐障害**: at-least-once・冪等 dedup・dead-letter、そしてプロセス再起動を越える永続化。長寿命ランタイムのクラッシュ検知と復元。
- **無音の失敗を作らない**: エラーも空返答も、必ず可視のフィードバックを返す。

### Kyberion の現状

- ホスト内 A2A は無条件 Bash・使い捨て署名鍵・単発 RPC・会話喪失。Mesh Hub は配送状態機械が実装済みだが駆動系が無く休眠。調整バスはメモリ内のみ。相関 ID は 3 つの独立キー空間に分断(調査: AA-01〜05、IL-02)。
- ハンドオフはペルソナ文字列交換のみ、A2A wire 契約が受入条件を落とす(調査: HO-01)。

### 取り込み(対応計画)

- **AA-01**(runtime 耐障害化)、**AA-02**(Mesh 配送ドライバ起動)、**AA-04**(会話モデル・バックプレッシャ)、**HO-01**(自己完結ハンドオフパケット)、**IL-02**(相関スレッド)、**UX-01**(無音失敗の解消)。

## 7. エージェントループ / ハーネス設計 — 決定論の骨格にモデル判断を挿す

### 原則

- **決定論的な制御フロー(ループ・条件・fan-out)はコードで、モデル判断はプロンプトで。** ハーネス(進め方)は再現可能なコードとして書き、その中の「判断が要るノード」だけをモデルに委ねる。「ステージ間に何を並列化し、何を検証し、何を統合するか」をコードで固定する。
- **ジャーナルからの決定論的レジューム**: 各ステップの実結果を記録し、再開時は「変わっていない接頭辞はキャッシュから即返し、最初に変わった箇所から生実行」。イベントは冪等に再適用。
- **モデル/エフォートをタスクごとに選ぶ**: 機械的横展開=軽量・低エフォート、標準実装=中位、設計判断/審判/敵対検証=上位・高エフォート。既定はセッションモデル継承、確信がある時だけ外す。
- **予算をハードシーリングにする**: トークン/コストの目標を制御として効かせ(プロンプト注入でなく)、超過で止める or 承認要求。深さを予算にスケールさせる。
- **無音の打ち切りを禁止**: top-N・no-retry・sampling で削ったら、何を落としたかを必ず log する。削ったのに「全部やった」と読めるのが最悪。

### Kyberion の現状

- ADF パイプライン(決定論)と mission worker(モデル駆動)はあるが、**「決定論骨格 + 判断ノード」を明示的に author する workflow-as-code の経路が無く**、JSON ADF のみ(vm での `core:transform` が唯一の脱出)。effort は adaptive ハードコード。予算は未強制(コスト集計すら直接 SDK 経路で欠落)。打ち切りはほぼ無音 slice(調査: HN 確認 §3/§4/§7、OP-01)。
- ジャーナルレジュームは MO-06 で計画済み(現状はバスがメモリ内、resume が儀式的)。

### 取り込み(対応計画)

- **HN-01**(モデル/エフォートのタスク単位選択と軽量モデル規律)、**HN-03**(workflow-as-code・並列/loop・無音打ち切り解消・effort 制御)、**MO-06**(journal レジューム)、**MO-05**(モデルルーティングの enforce)、**OP-01**(コスト計測・予算上限)。

## 8. 軽量モデル活用のチェックリスト(実務まとめ)

軽量モデル(haiku 級)にサブタスクを委ねる時、成果品質を保つための実務規律:

1. **スコープを 1 成果に絞る** — 「これ 1 つを作る/直す」。複数目的を 1 依頼に混ぜない。
2. **ブリフを自己完結にする** — 目的・制約・受入条件・返答形式・「迷ったら報告」を全部入れる。前提知識を要求しない。
3. **良質な例で seed する** — 最新でなく**最良**の先行成果物を手本として渡す(品質ランク再利用)。
4. **出力を構造で縛る** — schema-forced にして、検証してから受理。不一致は再プロンプト。
5. **検証を厳格化する** — 軽量モデルの成果ほど、動かす検証 + 敵対レビューを効かせる。
6. **パターン確立は上位モデルで** — 1 件目を上位で作り、その diff/パターンを軽量モデルの横展開に添付する。
7. **低品質なら escalate** — ルーブリックで poor なら上位モデルで再試行(HN-03/MO-07)。
8. **落としたものを log する** — バッチ・top-N で削ったら明示。

この 8 項目は本改善計画群のタスク割当方針(sonnet=パターン/判断、haiku=横展開/機械的、opus=設計/審判)の裏返しであり、Kyberion のサブエージェント委譲にそのまま適用できる。

## 9. まとめ — 何を Kyberion に足すか

Kyberion は**部品は揃っているが配線されていない**(品質機構・Mesh 配送・context pack・モデルルーティング・構造化出力がいずれも「実装済みだが未起動 or 限定領域」)。ハーネスモデルの取り込みとは、これらを「決定論の骨格に沿って正しく配線する」ことに他ならない。優先度順の要点:

1. **成果物評価の実配線**(MO-02/MO-07)— 完了主張を信用せず動かす + 休眠品質機構の起動。最も成果品質に効く。
2. **依頼と出力の構造化**(MO-04/HN-02)— context pack を配り、schema-forced で返させ検証。軽量モデル活用の前提。
3. **並列と改善ループ**(MO-03/HN-03)— 直列脱却と retry-until-quality/loop-until。
4. **モデル/エフォート/予算の実効化**(HN-01/MO-05/OP-01)— 機械的=安価、判断=強力を dispatch で。
5. **通信とハンドオフの自己完結化**(AA/HO/IL 系)— 相関貫通と聞き返し不要の引き継ぎ。
