# Kyberion アーキテクチャ改善提案: プロダクションレベルへの引き上げ

Kyberion を実験的フェーズから、大規模かつクリティカルなプロダクション運用に耐えうるシステムへと進化させるための、5 つの核心的なアーキテクチャ改善提案です。

## 1. Knowledge 基盤の再設計: 階層化・インデックス化と RAG 導入

### 現状の課題
現在 900 ファイルを超える `knowledge/` ディレクトリは、フラットまたは単純なディレクトリ構造であり、LLM が全容を把握するためのコンテキスト負荷が高い。また、必要な情報を動的に抽出する仕組み（RAG）が未整備である。

### 提案
- **セマンティック・インデックスの構築**: 知識層をベクトルデータベース（Chroma, Pinecone, あるいはローカルの pgvector 等）に同期し、ミッションの文脈に応じた関連知識のみを抽出する RAG (Retrieval-Augmented Generation) 機能をコアに統合する。
- **知識のグラフ化**: ファイル間の依存関係や参照関係を Mermaid や JSON で静的に管理するのではなく、ナレッジグラフとして抽出し、複雑な推論時にエージェントが「関連する法規」「関連する過去事例」を辿れるようにする。
- **Tier 間の物理的分離とプロキシ**: Tier（Personal/Confidential/Public）間の隔離を徹底するため、知識へのアクセスを抽象化された `KnowledgeService` 経由に限定し、アクセス権限の動的検証と PII マスキングを強制する。

## 2. Actuator のプラグイン化と分離

### 現状の課題
Actuator がモノリスなリポジトリ構造に含まれており、特定の実行環境（例：特定の OS や Python ライブラリが必要な場合）への配布や、サードパーティによる拡張が困難。

### 提案
- **プラグイン・アーキテクチャへの移行**: 各 Actuator を独立した `npm` パッケージまたは Docker コンテナ、WASM モジュールとして分離する。
- **オンデマンド・プル**: ミッション実行時に必要な Actuator がローカルにない場合、信頼されたレジストリから動的にダウンロード・検証・実行する仕組みを `capability_discovery.ts` に実装する。
- **サイドカー・モデル**: リソース消費の激しい Actuator（Browser, Media 解析等）を独立したプロセスまたはサイドカーコンテナとして実行し、gRPC や高速な IPC で通信することで、メインエンジンの安定性を確保する。

## 3. 統一観測基盤 (Observability) の構築

### 現状の課題
Trace や Audit のデータがミッションログやアーティファクトとして散在しており、全系を俯瞰したリアルタイム監視や、相関分析（例：LLM の応答遅延とトークン消費の相関）が困難。

### 1.5 提案
- **OpenTelemetry の全面採用**: エンジンの Phase 1 〜 5 の全ステップにスパンとイベントを埋め込み、分散トレーシングを実現する。
- **標準メトリクス・コレクタ**: `presence/` 層を拡張し、Prometheus 形式のメトリクスを露出。CPU/メモリ使用量、LLM コスト、ミッション成功率、Actuator の稼働状況を Chronos Mirror v2 で一元可視化する。
- **監査ログの不変性担保**: ガバナンスに関わる Audit Log を、ライトワン・リードメニー (WORM) 属性を持つストレージや、ハッシュチェーンによる改ざん検知が可能な分散レジャーに保存する。

## 4. 状態管理の堅牢化: ステートマシンの導入と冪等性担保

### 現状の課題
長寿命ミッションにおける Checkpoint/Resume がファイルベースの簡易的なものであり、複雑な並行処理や外部システムとの相互作用において、再開時の整合性を保つのが難しい。

### 提案
- **明示的なステートマシン定義**: 各ミッションのライフサイクルを ADF レベルで厳密な状態遷移図（Statechart）として定義し、不正な状態遷移をランタイムで拒否する。
- **イベントソーシング**: ミッションの状態を「現在のスナップショット」だけでなく、「発生したイベントのシーケンス」として記録。任意の時点へのリプレイ（Time Travel Debugging）を可能にし、診断能力を飛躍的に向上させる。
- **Idempotency Key の強制**: 全てのアクチュエータ操作（特に外部 API やファイル変更）において、ミッション ID とステップ ID を組み合わせた冪等性キーを必須とし、リトライ時の二重実行を防止する。

## 5. ガバナンスの自動化: ADF ベースのプログラマブル・ポリシー

### 現状の課題
Sudo Gate の判断ロジックがエージェントの推論に依存しており、組織固有のコンプライアンスルールやセキュリティポリシーを厳密かつ透過的に適用する仕組みが弱い。

### 提案
- **Policy-as-Code (PaC)**: OPA (Open Policy Agent) や Rego に類する、ADF 実行前にポリシーを静的解析するエンジンを統合する。
- **ADF ガードレール**: 特定の危険な命令（`rm -rf`, `curl` への秘密情報の埋め込み等）を、LLM の判断を待たずに正規表現や抽象構文木 (AST) 解析で遮断する。
- **動的権限昇格 (Just-in-Time Privileges)**: ミッションのコンテキストに応じて、一時的に特定のアクチュエータ権限をエージェントに付与し、完了後に即座に剥奪する最小権限原則 (PoLP) を自動化する。
