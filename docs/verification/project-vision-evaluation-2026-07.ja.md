# プロジェクト全体構想 評価レポート(2026-07-02)

> **評価対象**: Kyberion プロジェクトの構想全体 — テーゼ(WHY)、戦略(PRODUCTIZATION_ROADMAP)、アーキテクチャ原則(intent loop / mission control / data tiers)、およびそれらと実装の整合。
> **方法**: ビジョン・戦略文書の精読(WHY / README / PRODUCTIZATION_ROADMAP / INTENT_LOOP_CONCEPT / USER_EXPERIENCE_CONTRACT / GOVERNANCE / vision/\_default / mission-control-model / ecosystem-evolution-roadmap)+ ソースコード全体調査(9領域を並列調査。詳細な改善計画は [improvement-plans-2026-07](../developer/improvement-plans-2026-07/README.ja.md))。
> **既存評価との関係**: [PRODUCT_UX_EVALUATION_2026-05-29](../developer/PRODUCT_UX_EVALUATION_2026-05-29.ja.md)(UX 2.9/5、operator alpha 判定)、[evaluation_report](./evaluation_report.md)(100シナリオ理論評価 56%)、[architecture_recommendations](./architecture_recommendations.md) を前提とし、重複せず「構想そのもの」と「構想と実装の乖離」を評価する。なお evaluation_report の 56% は**紙上評価**であり実測ではない点に留意。

## 1. 総合評価(結論)

**構想は一貫して優れており、賭けの筋も良い。最大のリスクは構想の側ではなく、「構想が保証すると宣言していることを、実装がまだ保証していない」ギャップと、2つの製品人格の同居にある。**

| 観点                                             | 評価   | 一言                                                                          |
| ------------------------------------------------ | ------ | ----------------------------------------------------------------------------- |
| テーゼの明確さ(何であり何でないか)               | **A**  | 「organization work loop engine」・非ユーザーの明示は例外的に明確             |
| 戦略の現実性(OSS-first + FDE、Phase A〜D')       | **B+** | 旧 Phase C/D の削除・ゲート規律は誠実。KPI と第一優先ユーザーのずれが残る     |
| アーキテクチャ原則(intent loop / mission / tier) | **A−** | 「不可換の核 vs 可換の周辺」の切り分けは強い設計判断                          |
| 原則と実装の整合                                 | **C+** | 宣言済みの保証(tier 強制・trace・UX 契約)の執行機構に穴(§4)                   |
| 構想文書間の整合                                 | **B−** | 2人格問題・語彙増殖・tenant スコープの自己矛盾(§3)                            |
| 自己評価の誠実さ                                 | **A**  | D1〜D6 の 🔴 自己申告、ecosystem roadmap の過大主張の自己訂正は文化として貴重 |

## 2. 構想の強み

1. **「エンジンが資産、LLM は消耗品」という賭けの置き方**。`WHY.md:70-78` の「LLM は半年で置き換わる。それを統治された組織業務に変えるエンジンが生き残る」は、モデル間競争に巻き込まれない唯一の位置取りであり、INTENT_LOOP_CONCEPT の「6段ループの閉包のみ不可換、モデル・CLI・アクチュエータ・スキーマ名は可換」(`INTENT_LOOP_CONCEPT.md:96-104`)という具体的な設計原則にまで落ちている。構想→原則→(部分的に)実装まで一本の線が通っている。
2. **「誰のためでないか」が書いてある**(`WHY.md:52-56`)。チャット用途・turnkey SaaS 希望・コーディング特化は他所へ、という切り捨ては、スコープ膨張への最良の防波堤。§0「やらないこと」(SaaS化・マルチテナントGUI・公開API・OAuth/Stripe)を**削除の理由付きで**明記した PRODUCTIZATION_ROADMAP も同様に価値が高い。
3. **ミッション制御モデルの独自性**。「自由浮遊するエージェントを実行しない。ミッションを通じて実行する」・単一オーナー+リース・per-mission Git・decision-centric explainability(`agent-mission-control-model.md`)は、汎用エージェントフレームワークとの実際の差別化点で、24h+ 走行・監査という FDE 顧客の要求に直結する。
4. **自己評価とドッグフーディングの文化**。D1〜D6 を 🔴25%・🔴10% と自己申告し、ecosystem roadmap が自分の過大主張(「tier と ADF でリスクを構造的に排除」→「どちらも単独では不十分」`kyberion-ecosystem-evolution-roadmap-2026-06.md:34`)を訂正している。ロードマップの Phase ごとにミッションを切って distill する運用も実際に回っている(本評価の前提資料の多くがその産物)。
5. **段階ゲートの規律**。「未達なら次 Phase に入らない」(`PRODUCTIZATION_ROADMAP.md:404`)、3〜4週で外向きアウトプット、日英同格の CI 強制 — 進め方のルールが数値と結びついている。

## 3. 構想内部の緊張(文書 vs 文書)

1. **2つの製品人格の同居(最大の構造的緊張)**。公開文書(WHY/README)は謙抑的な OSS 開発者向け「委任エンジン」を語るが、`vision/_default.md` は「主権者の感情・美学」をタイブレーカーとする単一 Sovereign(創業者/CEO)向け自律意思決定パートナーを規定する。GLOSSARY も Sovereign 語彙を中核に置く。**KPI(GitHub Stars・contributor 数)は後者を測れず、前者の第一優先ユーザー「創業者・経営者」も測っていない**。どちらが主かを決めないと、UX(operator-surface の disconnected な質実さ vs chronos の "Sovereign Link")も文言もぶれ続ける。
2. **tenant 機構の「やらない/半分ある」矛盾**。§0 はマルチテナント GUI・RBAC を非目標と宣言する一方、POST_ONBOARDING_UX_ROADMAP は tenant 常時表示・tenant_slug・cross-tenant バナーを要求し、コードにも tenant 機構が半分実装済み。運用実態(単一組織内の複数テナント文脈)と「マルチテナント SaaS はやらない」は両立可能だが、**文書上の用語が同じ "tenant" なので矛盾に見える**。定義の分離(社内テナント文脈 vs SaaS マルチテナンシー)を GLOSSARY で行うべき。
3. **語彙の増殖 vs UX 契約の 4 概念原則**。USER_EXPERIENCE_CONTRACT は人間境界を「依頼・実行単位・成果物・次の一手」の4概念に制限するが、周辺文書は project/track/mission/task/packet/ledger/Charter/ADF… と概念を増やし続けている。契約はこのスプロールへの**訂正的な理想**として書かれており、実装はまだ追随していない(§4-3)。
4. **「声で5分の first win、依存ゼロ」の看板 vs 環境依存の現実**。UX 評価(2026-05-29)はブラウザ経路ですら OS 権限で落ちると報告済み。看板の強さと足元の脆さの差が最も大きい箇所で、Phase A のボトルネック(D1 🔴30%)と一致している。
5. **複数メンテナ前提のガバナンス文書 vs BDFL 実態**。GOVERNANCE.md の lazy consensus・7日ルールはまだ存在しないコミュニティ向けに書かれている。害は小さいが、K2(月間外部コントリビュータ≥5)達成までは「先に書いた憲法」であることを自覚しておくべき。
6. **経営者向け出力層が空洞**。第一優先ユーザーが経営者なのに、`CEO_STRATEGIC_REPORT.md` は `$(date)` が未展開のまま残る生成スタブで、経営向けの成果物ラインが実質未着手。evaluation_report でも Business & Strategy カテゴリは 30% で最下位タイ。

## 4. 構想と実装のギャップ(宣言された保証 vs 執行機構)

今回のコード全体調査で確認した、「構想文書が保証済みとして語る性質」と実装の乖離。**いずれも対応計画を作成済み**(括弧内)。

1. **「tier 分離は file-IO 境界で強制される」(`WHY.md:63`)** — 実際には、その強制装置である ESLint ルールがグローバル ignore により `libs/core`・`scripts`・`tests` で無効化されており、`libs/core` 内の native ドキュメントエンジン等 6 ファイルが raw `fs` で tier-guard を素通りしている(→ IP-01/IP-02)。
2. **「30日壊れない」(Phase B)** — CI は 589 テストファイル中 smoke 1本 + core のみ実行。actuators/scripts/契約・境界テスト約220本はどのワークフローでも走らず、カバレッジ閾値も未強制。長寿命サーバ(voice-hub)には未処理 Promise が残る(→ IP-03/IP-08)。
3. **「失敗が次の行動を教える」(D3)/ UX 契約** — 契約バリデータは実装・テスト済みだが呼び出しは 1 箇所・非ブロッキングのみ。ターミナル面は生 enum を露出し、明確化質問は英語固定、テキストブリッジ 4 面はエラー時に**無言**。D3 自己評価 🔴25% はコード実態と整合しており正確(→ UX-01/UX-03/UX-05)。
4. **「ADF は検証済み契約のみ実行、失敗は分類し修復」(AGENTS.md §1)** — その中核 `validateAndRepairAdf` にテストがゼロ。実トークンを消費する推論アダプタ 3 本も未テスト(→ IP-07)。
5. **「意図→計画→状態→結果を見せる」(OPERATOR_UX_GUIDE)** — ターミナルではパイプラインの開始と終了しか見えず、State が欠落している(→ UX-02)。
6. **再現性の足場のドリフト** — 壊れた npm script 11 個・存在しないパイプライン参照・スキーマ二重定義など、「決定論的に再実行できる」という価値提案の足場に参照切れが蓄積(→ IP-04/IP-12)。

**評価**: これらは構想の誤りではなく執行の未達であり、かつ大半が S〜M 規模で修復可能。裏を返せば、**構想文書が「already true」の時制で語る癖**(differentiators を現在形で列挙)が、外部読者に対して過大約束になっている。ロードマップ側は誠実に 🔴 を付けているので、WHY/README 側の時制を「設計原則(enforced by design, hardening in progress)」に揃えるだけで解消する。

## 5. 戦略への評価

- **OSS-first + FDE は依然正しい**(2026-05-29 評価と同意見)。今回の調査で加える論点は 2 つ:
  - **FDE 商売の信頼財は「ガバナンスが本当に効いていること」**。§4-1(tier 強制の穴)は機能バグではなく**商品の中核主張の穴**なので、Phase A の見栄え改善より優先度が高い。IP-01/02/03/07 を Phase B の受入条件に紐づけることを推奨。
  - **K1〜K6 に「第一優先ユーザー」の指標が無い**。経営者/パワーユーザーの委任成功を測る指標(例: 週次の委任タスク完遂数、承認往復の所要時間)を K 系に 1 つ追加しないと、OSS 健全性だけを最適化して本来のユーザー価値から漂流するリスクがある。
- **Phase 構成(A→B→C'→D')とゲート規律は妥当**。ただし現状ボトルネック(D1/D3/D5/D6)のうち D3(読める失敗)は本評価の UX-01/02/03 が直接の実装計画になっており、D5(1週間で貢献)には IP-03(CI が全テストを回す=外部貢献者が安心して PR を出せる)と IP-04/06(壊れた導線の除去)が効く。**ロードマップの D 系条件と improvement-plans の対応表を PRODUCTIZATION_ROADMAP 側に 1 行ずつ追記する**ことを推奨。

## 6. 提言(優先順)

1. **製品人格の主従を決める(構想判断・人間の意思決定)**: 「OSS 委任エンジンが主、Sovereign 経営パートナーは主要ユースケースの一つ」か、その逆か。決定を WHY.md に 1 段落で明文化し、GLOSSARY の Sovereign 語彙・chronos の文言・KPI をそれに追随させる。
2. **宣言と執行の同期(即時・実装)**: IP-01/02(tier 強制の実効化)、IP-07(ADF 修復のテスト)、IP-03(CI ゲート)を Phase B 受入条件に正式に紐づける。WHY/README の differentiator 節の時制を「enforced by design / hardening tracked in D1-D6」に修正する。
3. **D3(読める失敗)を UX-01→UX-02→UX-03 の順で潰す(実装)**: 無言の失敗の解消が最小コストで最大の体験改善。既存 UX 評価の UX-0(first-run 安定化)と並走可能。
4. **tenant 用語の二義性を GLOSSARY で解消する(小・文書)**: 「社内テナント文脈(実装済み・継続)」と「SaaS マルチテナンシー(非目標)」を別語にする。
5. **経営者向け出力ラインの最小実装(Phase B 後半)**: CEO_STRATEGIC_REPORT スタブを廃止または実データ接続し、K 系に経営者価値の指標を 1 つ追加する。
6. **既存評価の実測化(継続)**: evaluation_report(理論 56%)の主要シナリオ 10 本程度を、IP-03 で拡張した CI 上の実測スモークに昇格させ、「紙上の 56%」を実測値に置き換えていく。

## 7. 本評価の限界

- confidential tier(顧客固有計画)は tier 隔離のため参照しておらず、顧客文脈での構想妥当性は評価外。

## 追補: 競合比較 — OpenClaw / Hermes-Agent(2026-07-02 Web 調査)

2026 年のオープンソースエージェント市場は OpenClaw(旧 Clawdbot、Foundation 移行中・NVIDIA が企業向けラッパー NemoClaw を提供)と Hermes-Agent(Nous Research、2026-02 公開、3ヶ月で GitHub 140k stars・16+ メッセージングチャネル・自己改善スキルループ)の 2 強。

**勝てない土俵(競合が有意に強い)**: コミュニティ規模・スキルエコシステム・チャネル網羅・インストール体験。K1(GitHub Stars ≥1,000)はこの土俵での KPI であり、140k stars の相手に対して意味を持たない。

**Kyberion が有意に強い土俵**:

1. **ミッション第一級・組織業務モデル** — 両者は個人アシスタント/タスクループ中心で、OpenClaw は「承認ワークフロー・決定境界・コンプライアンスログのネイティブ機構が無い」と評される。per-mission Git・単一オーナーリース・decision-centric explainability を持つ製品は市場に見当たらない。
2. **データ tier 分離と customer オーバーレイ** — 両者のメモリは単一ユーザー空間。複数顧客・機密区分をエンジン層で分離する設計は FDE/SI 業態の中核要件で、競合の設計思想の外にある。
3. **統治された決定論パイプライン(ADF)** — Hermes の「自分でスキルを書いて成長する」モデルは供給網リスクの震源(悪性スキル 1,184 件検出、LiteLLM 供給網侵害、CVE 多発)として顕在化しており、preflight/検証済み契約のみ実行という Kyberion の逆張りは規制業種では優位。
4. **監査・規制適合を前提にした日本語第一級の FDE モデル** — Microsoft が OpenClaw の既定権限を「企業環境には過剰に寛容」と評し、NemoClaw のような「統制ラッパー」市場が生まれたこと自体が、Kyberion が最初から狙う層の需要の証明。

**戦略含意**: (a) 汎用個人アシスタントとして競合しない(チャネル網羅で追わず、OpenClaw/Hermes を surface/actuator として取り込む interop 戦略を検討)。(b) 優位性はすべて「統治が実際に効いていること」に依存するため、§4 の執行ギャップ(IP-01/02/03/07)解消が競争戦略そのものである。(c) 開発継続の価値判断: 汎用エージェント競争としては否、**規制業種向け governed mission layer + 社内(SBIグループ 6 テナント)実利用**としては肯。
