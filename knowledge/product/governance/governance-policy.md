# Dynamic Capability and Actuator Governance Policy v1.0

このポリシーは、Kyberion エコシステムにおいて、実行環境の能力（Capability）および統治方針（Policy）に基づき、アクチュエータと capability execution の実行を制御するための基準を定義する。

## 1. 動的能力検知 (Dynamic Capability Detection)

エージェントは、特定の OS に依存するだけでなく、その環境で必要なバイナリ（`say`, `screencapture` 等）が実際に利用可能かどうかを動的に判定しなければならない。

- **実装**: `libs/core/platform.ts` の `getCapabilities()` を使用する。
- **挙動**: 必要な能力が欠けている場合、capability execution は警告を表示してスキップするか、明確なエラーを投げて停止する。

## 2. Capability / Actuator 実行制限 (Capability Restriction)

特定の capability または actuator に対し、セキュリティやプロジェクトの方針に基づき、実行を明示的に禁止することができる。

- **管理ファイル**: `knowledge/product/governance/restricted-capabilities.json`
- **設定項目**:
    - `name`: capability または actuator 名。
    - `status`: `restricted`（制限中）または `active`（許可）。
    - `reason`: 制限の理由。
    - `allow_override`: 緊急時の上書き許可。

## 3. 実行時ガードレール (CLI Enforcement)

Kyberion CLI は、`run` コマンドの実行直前に以下のチェックを強制する。

1.  **Governance Check**: `restricted-capabilities.json` に記載があるか。
2.  **Platform Check**: actuator / capability の platform 制約と一致するか。
3.  **Capability Check**: 実行に必要な物理的コマンドが存在するか。

これら全ての条件を満たさない限り、対象の actuator / capability は物理的に読み込まれず、実行もされない。

## 4. 認可トラストバウンダリとLLMペイロードの非依存 (Authorization Trust Boundary)

システムの認可判定（例：`actorRole`、`tenantSlug` 等の決定）において、LLMが生成したツール呼び出し引数（`payload`）を信頼してはならない。

- **脅威モデル**: LLMは外部の非信頼データ（プロンプト・インジェクション等）の影響を受ける可能性があるため、LLMが動的に生成する JSON Payload は「任意の権限を詐称可能（Privilege Escalation）」な攻撃ベクトルと見なす。
- **実装原則**: `approval-gate` や意思決定ロジックなどのクリティカルな認可処理では、`payload.actor_role` のようなユーザー（LLM）からの申告値を使用せず、プロセスの実行環境コンテキスト（例: `process.env.MISSION_ROLE` や検証済みの `agentId`）から安全に権限を導出しなければならない。
- **環境変数の信頼性**: 環境変数はオーケストレーター等のインフラ層によってプロセス起動時に注入されるため、LLMのプロンプト・インジェクションでは改ざん不可能であり、セキュアなコンテキストとして扱うことができる。
