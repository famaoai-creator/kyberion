# ADF パイプライン検証計画

**Date**: 2026-05-04  
**Purpose**: `adf-pipeline-strategy-report.md` で整理した仮説を、実際のパイプライン実装に対して検証する

## 1. 検証目的

この計画の目的は、ADF パイプライン群を次の3分類で再評価することです。

1. 簡素化すべきもの
2. 抽象化すべきもの
3. 具象度を高めるべきもの

前提として、パイプラインはすでに「第一世代（シェル主導型）」から「第二世代（Reasoning 統合型）」へ移行しています。  
したがって検証では、単に `system:shell` を減らすだけでなく、**どこで意味レイヤーを増やし、どこで実行レイヤーを隠蔽するか**を判断します。

## 2. 検証仮説

### 2.1 簡素化仮説

対象:

- `trial-narrated-report.json`
- `meeting-facilitation-workflow.json`

仮説:

- 実行ロジックが ADF 内に埋め込まれすぎている
- 一時ファイル処理、バイナリ確認、環境前提チェックは専用操作へ切り出せる
- 簡素化により、保守コストと環境依存性が下がる

### 2.2 抽象化仮説

対象:

- `platform-onboarding.json`
- `faas-add-api.json`
- `design-from-requirements.json`

仮説:

- 「要件抽出 → 設計定義 → タスク分解」の SDLC パターンは共通化できる
- 共有 fragment に切り出しても、各パイプラインの個別性は維持できる
- 再利用化により、重複定義と修正漏れが減る

### 2.3 具象化仮説

対象:

- `ceo-strategic-report.json`
- `list-capabilities.json`

仮説:

- 出力構造が曖昧なため、LLM の揺らぎが出やすい
- Markdown セクションや必須データソースを明示すれば、再現性が上がる
- ここは抽象化よりも構造固定が有効

## 3. 検証方法

### 3.1 静的レビュー

各パイプラインについて次を確認する。

- `system:shell` の比率
- `reasoning:*` の使い方
- `ref` / 再利用フラグの有無
- ADF 内での責務分離
- 出力仕様の明示度

### 3.2 パターン分類

各対象を次の観点でラベル付けする。

- 実行手続きが主か
- 意味抽出が主か
- 共有化可能な骨格があるか
- 出力の構造固定が必要か

### 3.3 リスク確認

次をリスクとして確認する。

- 環境依存
- デバッグ難度
- 重複定義
- 生成物の不安定性
- テンプレート化による過剰抽象化

## 4. 成果物

検証完了時に出す成果物は次の3つです。

1. パイプライン分類表
2. 簡素化候補の具体的な削減案
3. 抽象化 fragment 候補と具象化テンプレート候補

## 5. 判定基準

### 5.1 簡素化を採用する条件

- ADF 本体の行数を減らせる
- 実行責務をアクチュエータへ追い出せる
- 再実行時の失敗原因が読みやすくなる

### 5.2 抽象化を採用する条件

- 3本以上のパイプラインに共通の手順がある
- 差分が目的だけで、実行骨格は共通化できる
- fragment 化しても読解コストが下がる

### 5.3 具象化を採用する条件

- 出力揺らぎが問題になっている
- 必須データソースが暗黙になっている
- Markdown / JSON などの構造を固定した方がレビューしやすい

## 6. 実施順序

1. `trial-narrated-report.json` と `meeting-facilitation-workflow.json` を先にレビューする
2. 次に `platform-onboarding.json` 系の SDLC 重複を確認する
3. 最後に `ceo-strategic-report.json` と `list-capabilities.json` の構造固定を確認する

## 7. ミッション化の判断

この検証は、次のいずれかに該当した場合はミッション化する。

- 3群以上の対象にまたがる
- 実装変更が必要になる
- 共有 fragment の設計判断が必要になる
- 形式固定のテンプレート設計が必要になる

逆に、静的レビューだけで結論が出る場合は、ミッション化せずにレビュー文書として終える。
