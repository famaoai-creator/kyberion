# ADF パイプライン全体評価レポート

**Date**: 2026-05-04  
**Scope**: ADF パイプライン群の設計成熟度、抽象化方針、簡素化余地の評価

## 1. 総評

現在のパイプライン群は、「第一世代（シェル主導型）」から「第二世代（Reasoning 統合型）」への移行期にあります。

初期のパイプラインは `system:shell` を多用し、複雑なロジックを無理やり JSON 内に記述しています。一方で、最近追加されたものは `reasoning:*` を活用して高度な判断をエージェントに委ねる、より洗練された抽象化へ進んでいます。

この変化は方向として正しいです。次の課題は、すべてを一律に抽象化することではなく、次の3つを切り分けることです。

- 簡素化すべきもの
- 抽象化すべきもの
- 具象度を高めるべきもの

## 2. 評価と改善の方向性

### 2.1 簡素化すべきもの

**対象**:

- `trial-narrated-report.json`
- `meeting-facilitation-workflow.json`

**課題**:

- `system:shell` 内で `cat <<'EOF'` や複雑な test コマンドが多用されており、デバッグが困難
- 具象度が高すぎて、環境変化に弱い
- 例: シェルバイナリ差、実行環境差、テストランナー差

**評価**:

これらは「何をしたいか」より「どうやっているか」が前面に出すぎています。結果として、ADF 本来の意味レイヤーが薄まっています。

**改善案**:

- 一時ファイル作成
- バイナリチェック
- 事前条件検証

これらを専用アクチュエータ・オペレーションへ追い出すべきです。

例:

- `system:ensure_binaries`
- `system:prepare_temp_workspace`
- `system:assert_runtime_prereqs`

これにより、パイプライン定義を大幅に短縮できます。

### 2.2 抽象化すべきもの

**対象**:

- `platform-onboarding.json`
- `faas-add-api.json`
- `design-from-requirements.json`

**課題**:

- 「要件抽出 → 設計定義 → タスク分解」という SDLC の共通パターンが複数ファイルで重複している
- それぞれが個別の具象パイプラインとして実装されている

**評価**:

これは最も抽象化効果が高い領域です。個別パイプラインの差分は残しつつ、共通骨格は再利用可能な部品として切り出すべきです。

**改善案**:

- `pipelines/fragments/standard-sdlc-loop.json` として抽出
- 各パイプラインからは `ref` オペレーションで呼び出す
- 変更点は各パイプラインの目的差分だけに限定する

### 2.3 具象度を高めるべきもの

**対象**:

- `ceo-strategic-report.json`
- `list-capabilities.json`

**課題**:

- 成果物形式が LLM 任せになっている
- どのナレッジを優先的に読むかが曖昧
- Markdown の構造が安定しないリスクがある

**評価**:

ここは逆に、抽象化しすぎない方が良いです。出力の安定性、再現性、レビュー容易性を優先する必要があります。

**改善案**:

- 期待する Markdown の見出し構造を明示する
- 必須データソースを `params` 側で指定する
- `VISION.md` などの必須参照を固定化する
- 構造化プロンプト・テンプレートを ADF 内に持たせる

## 3. 実証済みデザインパターン

検証を通じて、以下の3つを今後の ADF 作成の標準パターンとすることを推奨します。

### 3.1 ガード・ファースト

機密情報を扱う際は、最初のステップで必ず `mission_tier` をチェックする。

### 3.2 エビデンス・ファースト

推論させる前に、必ず `system:collect_artifacts` で現実の状態をキャプチャし、コンテキストに注入する。

### 3.3 冪等シェル

どうしてもシェルを使う場合は、`mkdir -p` や `|| echo 'not_found'` を使い、何度実行しても止まりにくい形にする。

## 4. 結論

Kyberion のパイプラインは、「コードとしてのパイプライン」から「意図としてのパイプライン」へ進化すべきです。

具象的なシェル操作はアクチュエータへ隠蔽し、ADF には「何を入力し、何を判断し、何を生成するか」という高レイヤーな意味だけを残すのが理想です。

## 5. 推奨アクション

1. `trial-narrated-report.json` と `meeting-facilitation-workflow.json` を簡素化対象として分解する
2. `platform-onboarding.json` 系の共通 SDLC 骨格を fragment 化する
3. `ceo-strategic-report.json` と `list-capabilities.json` の Markdown 構造を固定化する
4. その前に、対象パイプライン群が本当に同じ問題を持つかを、分類軸付きで再検証する
