# The Data Ingestion Protocol (外部データ持ち込み規約)

このドキュメントは、エコシステム（サンドボックス）外部のデータを安全かつ統制された形で内部に取り込むための唯一の公式ルール（Ingestion Vectors）を定義する。
`tier-guard.cjs` による厳格なサンドボックス境界を維持しつつ、実用的なデータ連携を実現するためのプロトコルである。

## 1. 境界防衛の原則 (The Sandbox Principle)

エコシステムのルートディレクトリ外にあるファイルシステム、および未認可の外部ネットワークへの直接アクセスは、システムレベル（`tier-guard.cjs`）で物理的に遮断される。
外部データは、以下の4つの「正規のベクトル（Ingestion Vectors）」のいずれかを経由してのみ、エコシステム内に持ち込むことができる。

## 2. Ingestion Vectors (データ持ち込みの4つのルート)

### Vector 1: Manual Vaulting (主権者による物理的持ち込み)
- **概要**: ユーザー（主権者）自身が、ローカル環境や外部のファイルを `vault/` ディレクトリにコピーして配置する手法。
- **対象データ**: レガシーコード、PDFドキュメント、データベースのダンプファイル、アクセス権のないログファイルなど。
- **ポリシー**: 全ての「未精査の生データ（Raw Data）」は `vault/` に格納されなければならない。AIエージェントは `vault/` の内容を**読み取り専用**として解釈し、必要なエッセンスのみを抽出して `knowledge/` へ「蒸留（Distill）」する。

### Vector 2: Connector Skills (API経由のシステム間連携)
- **概要**: 事前に認可されたクレデンシャルを用いて、専用のコネクタスキル（例: `backlog-connector`, `github-connector`）が外部SaaS等からデータを取得する手法。
- **対象データ**: チケット情報、リモートリポジトリのPR、Slackのメッセージなど。
- **ポリシー**: クレデンシャルは必ず `knowledge/personal/connections/` に保存され、スキル固有のスコープ（`tier-guard`のアクセス制御）に従って読み出されなければならない。取得したデータは、一時的な解析のために `scratch/` や `vault/` に出力されるか、メモリ上で直接処理される。

### Vector 3: Agentic Web Fetching (公開ウェブ情報の取得)
- **概要**: `api-fetcher` などのスキルや、AIの組み込みツール（`web_fetch` 等）を用いて、認証不要の公開URLから情報をスクレイピング・取得する手法。
- **対象データ**: 公式ドキュメント、オープンソースのライセンス情報、公開APIのレスポンスなど。
- **ポリシー**: 取得対象は公開情報（Public Data）に限定される。取得したデータは「外部の知恵（External Wisdom）」として扱われ、AIによる検証と構造化を経て `knowledge/` に取り込まれる。

### Vector 4: Controlled Symlinking (厳格管理されたシンボリックリンク)
- **概要**: ローカルの巨大なプロジェクトディレクトリ等を `vault/` 配下にシンボリックリンクとして接続する手法。
- **対象データ**: ギガバイト単位の巨大なログファイルや、リアルタイムで変更される外部のビルドディレクトリ。
- **ポリシー**: セキュリティエスケープのリスクがあるため**原則非推奨**。使用する場合は、対象が「読み取り専用」であることを主権者が保証した上で、`tier-guard.cjs` がシンボリックリンクの解決先を安全とみなす範囲でのみ運用される。

## 3. The Distillation Mandate (蒸留の義務)

どのVectorを経由してデータが持ち込まれた場合でも、生データ（Raw Data）をそのまま `knowledge/` や `active/projects/` に書き込んではならない。
エージェントは必ず**「情報の抽出・構造化・秘匿情報のマスキング」**という蒸留プロセスを実行し、純化された知恵（Intel）のみを内部のナレッジベースに永続化しなければならない。
